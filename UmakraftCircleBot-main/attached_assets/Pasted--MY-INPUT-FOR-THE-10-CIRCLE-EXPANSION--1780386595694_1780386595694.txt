commands for Circle 2–10.

The direction you've already taken is the correct one:

One codebase
One command set
One scheduler
One database

with:

WHAT I WOULD KEEP
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

All circles should continue using:

/fan_gain
/leaderboard
/joindate
/link
/set_fans
/status
milestone system
attendance system

The command should determine the circle internally.

Example:

User runs /joindate

Bot flow:

search Circle 1
search Circle 2
search Circle 3
...
search Circle 10

first match wins

No:

/joindate_circle2
/joindate_circle3
/fan_gain_circle7

That becomes impossible to maintain.

BIGGEST DESIGN RULE
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Every production function should eventually look like:

someFunction(circleId, ...)

instead of:

someFunction(...)

Examples:

buildSnapshot(circleId)
getMembersForCircle(circleId)
upsertMemberForCircle(circleId)
syncCircleData(circleId)

This is the pattern your recent fixes are already moving toward.

Whenever I see:

config.circleId

inside runtime logic,

I would immediately ask:

"Should this actually be circleId?"

WHAT I WOULD AVOID

Do not make:

circle1_members
circle2_members
circle3_members

style tables.

Your current design is much better:

members
circle_id
trainer_id
...

Composite keys scale cleanly.

The same applies to:

daily_gains
attendance
milestone_fired

Keep them shared with circle_id separation.

WHAT MAY BECOME YOUR NEXT PROBLEM
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Not commands.

Not storage.

Snapshots.

Currently every circle eventually requires:

fetchCircle()
buildSnapshot()
classifyMembers()
computeMemberStats()

With:

2 circles

this is trivial.

With:

10 circles
300 members each

snapshot generation becomes the expensive part.

I would begin thinking about:

cache per circle
snapshot timestamp per circle
stale detection per circle

before adding Circle 5+.
