# Fan Gain Blueprint

## Purpose

This blueprint defines the visual and data structure for the `/fan_gain` deliverable.

It describes how a fan gain report should be assembled, presented, and delivered after the command requests it.

## Product overview

The `/fan_gain` report summarizes how a trainer's fan count has changed over time and highlights recent gains, trends, and important flags.

The report is intended for quick review and monitoring, with an emphasis on daily, weekly, and monthly growth.

## Layout

1. Header
   - Report title: `Fan Gain Report`
   - Trainer name / ID
   - Timestamp
   - Status badge
2. Summary metrics band
   - `dailyGain`
   - `weeklyGain`
   - `monthlyGain`
   - `rankChange`
3. Trend section
   - sparkline or bar chart for recent fan history
   - optional “gain pace” annotation
4. Flags and notes panel
   - active flags (threshold breaches, rank drop, large gains)
   - short contextual note
5. Footer
   - blueprint version
   - source metadata
   - generated by UmaKraft command

## Component definitions

### Header

- `title`: string
- `trainerName`: string
- `trainerId`: string
- `generatedAt`: ISO timestamp
- `status`: enum(`OK`, `Warning`, `Alert`)

### Summary metrics

- `dailyGain`: number
- `weeklyGain`: number
- `monthlyGain`: number
- `rankChange`: number

### Trend section

- `trendPoints`: array of { `label`: string, `value`: number }
- `trendType`: enum(`sparkline`, `bar`, `line`)
- `trendNote`: string

### Flags and notes

- `flags`: array of {
  - `code`: string
  - `label`: string
  - `severity`: enum(`low`, `medium`, `high`)
  - `description`: string
  }
- `note`: string

### Footer

- `blueprintVersion`: string
- `source`: string
- `command`: `/fan_gain`

## Data contract

The blueprint expects a payload with these sections:

- `meta`
  - `trainerId`
  - `trainerName`
  - `generatedAt`
  - `status`
- `metrics`
  - `dailyGain`
  - `weeklyGain`
  - `monthlyGain`
  - `rankChange`
- `trend`
  - `points`: array of { `label`, `value` }
- `flags`
  - array of flag objects
- `notes`
  - `summary`

## Workflow

```text
Discord User
      │
      ▼
   /fan_gain
      │
      ▼
Command validates request
      │
      ▼
Distribution/Retriever fetches approved fan gain product
      │
      ▼
Delivery renders the report using the Fan Gain blueprint
      │
      ▼
Discord Response
```

## Validation

A blueprint implementation must ensure:

- metrics are present and numeric
- trend points exist
- flags include required fields
- the command metadata matches `/fan_gain`

## Notes

- `/fan_gain` should be treated as a request for a fan gain product, not a calculation engine.
- The blueprint is responsible for presentation, while the Refinery and Compiler supply the computed values.
