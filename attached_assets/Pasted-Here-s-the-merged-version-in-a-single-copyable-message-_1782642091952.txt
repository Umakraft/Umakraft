# /profile System Enhancement, Historical Synchronization & Discord Avatar Integration

## Objective

Enhance the existing `/profile` command without changing its current architecture.

The existing implementation already supports:

* Profile Image Report
* Achievements
* Milestones
* Completion Streaks
* Attendance Streaks
* Rankings
* Honors
* Personal Records
* Progress to Next Milestone
* Multi-circle Support
* Historical Backfill Trigger

The goal of this milestone is to transform `/profile` into the permanent historical identity system for every trainer.

---

# Design Philosophy

The local database must become the permanent source of truth.

uma.moe is used only as the external data provider.

Historical data should be imported once, stored permanently, and reused by every future `/profile` request.

The `/profile` command itself should never perform expensive historical requests.

---

# Initial Historical Import

When a trainer is linked for the first time or when historical data is missing:

1. Detect the earliest available month on uma.moe.
2. Import every available month in chronological order.
3. Store every month's data locally.
4. Automatically calculate:

   * Milestones
   * Achievements
   * Completion Streaks
   * Personal Bests
   * Lifetime Fan Gain
   * Historical Monthly Statistics
5. Mark the trainer as fully synchronized.

This is a one-time operation.

---

# Monthly Automatic Synchronization

After the monthly tally has completed, the bot should wait **one day** before synchronizing to ensure the final monthly results on uma.moe are stable.

Workflow

```text
Monthly Tally Ends
        ↓
Wait 1 Day
        ↓
Fetch Latest Monthly Results
        ↓
Append New Month To Local Database
        ↓
Update Milestones
        ↓
Update Achievements
        ↓
Update Completion Streaks
        ↓
Update Personal Records
        ↓
Update Lifetime Statistics
        ↓
Mark Month As Synchronized
```

Only the newest month should be imported.

Previously synchronized months must never be processed again.

---

# Synchronization Protection

Create a synchronization tracking table.

```text
profile_month_sync

trainer_id
year
month
status
processed_at
```

Status values

* pending
* syncing
* completed
* failed

If a month is already marked as completed, skip processing.

---

# Restart Safety

The synchronization system must be restart-safe.

```text
Restart
    ↓
Read Synchronization Table
    ↓
Resume Incomplete Trainers
```

Requirements

* No duplicated history
* No duplicated milestones
* No duplicated achievements
* No duplicated personal records
* Resume exactly where synchronization stopped

---

# /profile Data Source

The `/profile` command should read only from the local database.

Never fetch historical information directly from uma.moe during profile generation.

Benefits

* Instant profile generation
* Reduced requests to uma.moe
* Stable statistics
* Consistent milestone history
* Works even if uma.moe is temporarily unavailable

---

# Discord Avatar Integration

The profile should automatically display the linked Discord member's current avatar.

Avatar priority:

1. Linked Discord user's custom avatar
2. Discord default avatar
3. Bot placeholder image (only if the Discord user cannot be resolved)

The avatar should **refresh every time `/profile` is generated**, ensuring it always reflects the member's latest Discord profile picture.

Do **not** permanently store avatar images in the database.

Only historical profile statistics should be stored locally.

# Final Goal

The `/profile` command should become the permanent identity card of every trainer.

Every generated profile should immediately identify the member through their **current Discord avatar** while showcasing:

* Who the trainer is
* Their complete historical journey
* Every milestone earned
* Every achievement unlocked
* Their progression through time
* Their consistency
* Their personal records
* Their contribution to the circle

After the initial historical import, the profile should remain automatically updated through the monthly synchronization process, which runs once on the day after each tally period.

The result should be an extremely fast, visually polished, and fully data-driven profile system that permanently preserves every trainer's history while minimizing requests to uma.moe.
